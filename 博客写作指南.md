# 博客写作指南

> 本文回答一个核心问题：**日常在哪里编辑、怎么编辑这个博客。**

## 〇、总原则：一切编辑都在本地文件夹

你的博客源码就是 `F:\MyProjects\MyBlog` 这个文件夹。

- **编辑器**：用任何你顺手的工具打开整个文件夹即可——VS Code、JetBrains IDE、Typora、甚至记事本。推荐 VS Code 或 Typora（Markdown 写作体验好）。
- **发布方式**：本地改完 → `git push` → GitHub Actions 自动构建部署 → 约 1 分钟后网站更新。
- **你永远不需要碰** `public/`（构建产物，本地预览生成的，已被 git 忽略）和 `themes/PaperMod/`（主题，不要直接改里面的文件）。

日常只和这几个地方打交道：

```
F:\MyProjects\MyBlog\
├── content\posts\tech\        ← 技术文章写在这
├── content\posts\humanities\  ← 人文文章写在这
├── content\about\index.md     ← 关于页
├── static\images\             ← 全局图片（头像等）
└── hugo.yaml                  ← 站点配置（改菜单、标题等）
```

## 一、添加一篇新文章

### 方法 A：命令行（推荐，自动套模板）

在项目根目录打开终端（PowerShell），运行：

```powershell
# 技术文章
hugo new posts/tech/redis-notes/index.md

# 人文随笔
hugo new posts/humanities/du-shu-bi-ji/index.md
```

这会生成 `content\posts\tech\redis-notes\index.md`，并自动填好 front matter 模板。

> 文件夹名就是文章的 URL（如 `/posts/tech/redis-notes/`），建议用英文或拼音，不要用中文和空格。

### 方法 B：手动复制

直接复制一篇现有文章的文件夹（如 `hello-hugo`），改名，然后编辑里面的 `index.md`。

### 写作要点（front matter）

每篇文章开头的 `---` 之间是文章元信息：

```yaml
---
title: "Redis 学习笔记（一）：数据结构"   # 文章标题
date: 2026-07-05T10:00:00+08:00        # 发布时间（未来时间不会显示！）
draft: false                            # ⚠️ 必须改成 false 才会发布
summary: "一句话摘要，显示在首页列表"
categories: ["tech"]                    # tech 或 humanities，决定文章归入哪个栏目
tags: ["Redis", "数据库"]               # 标签，随意写，可多个
cover:
  image: "images/cover.png"             # 封面图（可选，删掉整个 cover 块则无封面）
  relative: true
---
```

**最容易踩的两个坑**：
1. `draft: true` 的文章不会出现在线上（本地 `hugo server -D` 能看到）；
2. `date` 是未来时间的文章也不会显示。

### 插入图片

把图片放进**文章自己的 `images\` 文件夹**（和 `index.md` 同级，没有就新建一个），然后：

```markdown
![图片说明](images/screenshot.png)
```

### 插入图表（流程图 / 架构图 / 时序图）

不用作图软件，直接写 mermaid 代码块，网页上自动渲染成图：

````markdown
```mermaid
graph LR
    A[浏览器] --> B[Tomcat] --> C[Controller]
```
````

语法参考 https://mermaid.js.org ，支持流程图（graph）、时序图（sequenceDiagram）、
类图、状态图等。

### 插入视频

```markdown
{{</* bilibili BV1GJ411x7h7 */>}}     ← B 站，传 BV 号
{{</* youtube dQw4w9WgXcQ */>}}       ← YouTube，传视频 ID
```

## 二、修改 / 删除一篇文章

- **修改**：直接编辑对应的 `index.md`，保存后 push 即可。
- **删除**：把整个文章文件夹删掉（如删除 `content\posts\tech\redis-notes\` 整个目录），push 后线上就消失了。
- **暂时下架不删稿**：把 front matter 的 `draft` 改回 `true`，文章会从线上消失但稿子还在。

## 三、新增一个栏目

栏目 = 分类（category）+ 导航菜单项。比如想加一个「生活记录」栏目：

**第 1 步**：在 `hugo.yaml` 的 `menu.main` 里加一项：

```yaml
menu:
  main:
    - name: 生活记录
      url: /categories/life/
      weight: 3        # weight 决定菜单顺序，数字小的在前
    # ...其他菜单项，注意调整后面几项的 weight
```

**第 2 步**：（可选）新建对应的内容目录 `content\posts\life\`，方便文件归类。

**第 3 步**：写文章时 front matter 设 `categories: ["life"]`。

只要有至少一篇文章带这个分类，`/categories/life/` 页面就会自动生成，不需要手动创建页面。

## 四、写系列文章（专题连载）

想围绕一个主题写多篇连载（如「JDK21 与 Spring 实战」），用 **series（系列）** 机制：

**第 1 步**：文件上，给系列建一个子目录，每篇文章按序号命名：

```
content\posts\tech\jdk21-spring\
├── 01-jdk21-features\index.md      # 第一篇：JDK21 核心特性
├── 02-spring-version\index.md      # 第二篇：Spring 版本选择
└── 03-architecture\index.md        # 第三篇：架构设计……
```

**第 2 步**：每篇文章的 front matter 里标注**同一个系列名**（必须一字不差）：

```yaml
categories: ["tech"]                    # 栏目照常
tags: ["Java", "Spring Boot"]           # 标签照常
series: ["JDK21 与 Spring 实战"]        # ← 系列标注
```

**第 3 步**：没有第 3 步。Hugo 会自动生成系列聚合页
`/series/jdk21-与-spring-实战/`，读者点进去能看到整个系列的文章列表。
所有系列的总览在 `/series/`。

此外，**栏目页（如「技术笔记」）会自动按系列分组展示**：系列连载显示为一个
目录卡片（按阅读顺序列出各篇），不属于任何系列的文章归入「独立文章」区。
这个逻辑在 `layouts/categories/term.html`，无需你做任何事，加了 `series`
标注的文章会自动进入对应分组。

**几个实用建议**：

- 标题里带上序号（如「JDK21 与 Spring 实战（一）：……」），系列页按日期倒序排列，
  序号能帮读者找到阅读顺序；
- 可以先把整个系列的骨架文章都建好（`draft: true` 占位，写好大纲注释），
  再逐篇填充发布——本仓库 `content\posts\tech\jdk21-spring\` 下就有现成的骨架示例；
- 在每篇结尾加"下一篇预告"和上一篇的链接，串联阅读体验；
- 系列名如果想要干净的英文 URL，就用英文系列名（如 `series: ["jdk21-spring"]`），
  中文系列名的 URL 会包含中文字符（能正常访问，只是链接看起来长）。

## 五、修改站点本身

都在 `hugo.yaml` 里，常用的：

| 想改什么 | 改哪里 |
| --- | --- |
| 网站标题 | `title` |
| 首页问候语 | `params.homeInfoParams` 的 `Title` / `Content` |
| 社交图标 | `params.socialIcons`（支持 github、email、bilibili、x 等） |
| 导航菜单 | `menu.main` |
| 代码高亮风格 | `markup.highlight.style`（如 monokai、dracula、github-dark） |

关于页则直接编辑 `content\about\index.md`。

## 六、标准工作流（每次都一样）

```powershell
# ① 写作/修改/删除（用编辑器操作 content\ 下的文件）

# ② 本地预览，浏览器打开 http://localhost:1313 检查效果
hugo server -D -F       # -D 连草稿一起预览，-F 连未来日期的文章也预览；Ctrl+C 退出

# ③ 确认没问题后，发布
git add -A
git commit -m "post: 新增 Redis 学习笔记"
git push
```

push 后打开仓库的 **Actions** 页可以看构建进度，绿色 ✅ 后约半分钟，
`https://leopardstack.github.io/` 即更新。

> 💡 也可以直接在 GitHub 网页上编辑：打开仓库 → 找到 `content\` 下的 md 文件 →
> 点铅笔图标编辑 → Commit changes，效果等同于本地 push。适合在别的电脑上改错别字，
> 但注意之后回到本机要先 `git pull` 同步。

## 八、换电脑 / 本机坏了怎么办

**先放心：只要你 push 过，博客的全部内容（文章、图片、配置、部署流程）都完整存在
GitHub 仓库里，本地机器坏了不丢任何东西。** 网站本身也不受影响——它跑在 GitHub
Pages 上，和你的电脑无关。

在任何一台新电脑上恢复写作，只需四步：

### 第 1 步：装两个软件

- **Git**：https://git-scm.com/downloads
- **Hugo（必须是 extended 版）**：
  ```powershell
  winget install Hugo.Hugo.Extended     # Windows
  ```
  ```bash
  brew install hugo                      # macOS（brew 装的默认就是 extended）
  ```
  装完用 `hugo version` 验证，输出里应包含 `+extended` 字样。

### 第 2 步：配置 Git 身份

```powershell
git config --global user.name "Leopard"
git config --global user.email "你的GitHub邮箱"
```

### 第 3 步：克隆仓库（⚠️ 必须带 --recursive）

```powershell
git clone --recursive https://github.com/LeopardStack/LeopardStack.github.io.git MyBlog
```

`--recursive` 是为了把主题（submodule）一起拉下来，忘了加会导致本地预览一片空白。
补救命令：`git submodule update --init --recursive`。

### 第 4 步：验证后照常写作

```powershell
cd MyBlog
hugo server -D        # 打开 http://localhost:1313，能看到博客就恢复完成
```

之后的工作流和原来完全一样。第一次 `git push` 时会弹出 GitHub 登录窗口，
用浏览器授权一次即可。

### 两个好习惯，让"电脑坏了"永远无痛

1. **写完就 push**——只有"写了但没 push"的内容才会随电脑丢失。哪怕文章没写完，
   也可以保持 `draft: true` 先 push 上去存着（草稿不会出现在线上）。
2. **多台电脑切换时，开工前先 `git pull`**——把另一台电脑推送的最新内容同步下来，
   避免两边改乱产生冲突。

## 九、出问题了怎么办

| 现象 | 大概率原因 |
| --- | --- |
| 文章 push 了但线上看不到 | `draft: true` 没改，或 `date` 是未来时间 |
| 图片不显示 | 路径大小写不对，或图片没放进文章的 `images\` 目录 |
| Actions 构建失败（红 ❌） | 点进失败日志看报错；常见是 front matter 的 yaml 格式错误（冒号后少空格、引号不配对） |
| 本地预览正常线上样式全丢 | `hugo.yaml` 里 `baseURL` 被改错了 |
