# AI 写作提示词模板

> 用法：复制对应模板，把【方括号】里的内容替换成你的实际需求，发给 Claude Code / Codex。
> 项目规范已写在仓库根目录的 `CLAUDE.md`（Claude Code 自动读取）和 `AGENTS.md`（Codex 自动读取），
> 所以提示词里不需要重复解释目录结构和 front matter 格式。

---

## 模板 1：新增技术文章（最常用）

```text
请为我的博客新增一篇技术文章。

主题：【Redis 持久化机制：RDB 与 AOF 对比】
目标读者：【有一定后端基础，但没深入用过 Redis 的开发者】
文章要点（按此展开，可以适当补充）：
1. 【为什么需要持久化】
2. 【RDB 的原理、配置、优缺点】
3. 【AOF 的原理、配置、优缺点】
4. 【生产环境怎么选：混合持久化】

要求：
- 篇幅约【2000】字，代码示例必须真实可运行，配置项给出具体版本（【Redis 7.x】）
- 我提供的素材：【无 / 见 F:\xxx\笔记.md，以我的笔记为准展开】
- 需要配图的位置：先用文字占位「（图：xxx示意）」标出，不要生成假图片
- 写完后本地构建验证，保持 draft: true，等我预览确认后再改 false
- 不要 push，我自己检查后发布
```

## 模板 2：新增人文随笔

```text
请为我的博客新增一篇人文随笔。

主题：【读《乡土中国》——差序格局在今天的互联网社区里还成立吗】
我的核心观点/想法（务必以此为骨架，不要替我发明观点）：
【1. 差序格局描述的是以己为中心的关系网络
 2. 我观察到微信群和豆瓣小组呈现两种不同的组织形态
 3. 我的结论是：技术没有消灭差序格局，只是给了它新的载体】
素材：【我的读书笔记片段粘贴在下面 / 无】

要求：
- 篇幅约【1200】字，散文随笔风格，第一人称
- 语气克制、具体，多用我给的例子，禁止空泛的大词和排比抒情
- 引用原书内容要准确，不确定的引文标注「（原文待核）」
- 保持 draft: true，不要 push
```

## 模板 3：润色 / 修改已有文章

```text
请帮我润色这篇文章：content/posts/【tech/redis-notes】/index.md

修改目标：【逻辑更顺 / 压缩到 1500 字以内 / 补充 xx 部分 / 修正技术错误】
保持不变的部分：【我的观点和例子 / 代码块 / 整体结构】
改动方式：直接修改文件，但改动幅度大的段落在回复里说明改了什么、为什么改。
```

## 模板 4：系列文章规划

```text
我想写一个【Java 并发编程】系列，预计【6-8】篇。
请先给我一份系列大纲（每篇的标题、核心内容、篇幅预估、由浅入深的顺序），
我确认大纲后，再让你逐篇写作。目标读者：【用过多线程但没系统学过 JUC 的人】。
先只出大纲，不要开始写文章。
```

## 模板 5：配图与排版检查（发布前）

```text
请检查 content/posts/【tech/redis-notes】/ 这篇文章：
1. front matter 是否符合本站规范（date 时区、categories、summary）
2. 所有图片引用的文件是否真实存在于 images/ 目录
3. 代码块是否都标注了语言
4. 运行 hugo 构建确认无 ERROR
5. 列出所有仍是占位状态的配图，告诉我需要准备哪些图
都通过的话把 draft 改为 false，但不要 push。
```

---

## 使用建议

1. **素材越多，质量越高。** AI 最擅长的是把你零散的笔记、观点、代码整理成结构化文章；
   最危险的是让它凭空写它"知道"的东西（容易一本正经地编造）。技术文优先喂你自己验证过的代码，
   人文文务必给出你自己的观点。
2. **保持 draft → 预览 → 发布的节奏。** 模板里都写了"保持 draft: true、不要 push"，
   预览确认没问题后，一句「把 draft 改成 false 并 push」即可发布。
3. **一次一篇。** 不要让 AI 一口气生成五篇文章，逐篇打磨的质量远高于批量生产。
4. **规范改了要同步。** 如果以后新增栏目、改了 front matter 字段，记得同步更新
   `CLAUDE.md` 和 `AGENTS.md`，否则 AI 会按旧规范干活。
